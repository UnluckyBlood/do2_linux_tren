## Part 3. Утилита iperf3

### 3.1. Скорость соединения

Перевод единиц измерения скорости передачи данных:

- **8 Mbps в MB/s**:  
  `8 Mbps = 8 / 8 = 1 MB/s` (поскольку 1 байт = 8 бит).

- **100 MB/s в Kbps**:  
  `100 MB/s = 100 * 8 = 800 Mbps = 800 * 1000 = 800 000 Kbps` (в десятичной системе, где 1 Mbps = 1000 Kbps).

- **1 Gbps в Mbps**:  
  `1 Gbps = 1000 Mbps`.

### 3.2. Утилита iperf3

Измерение скорости соединения между ws1 и ws2:

- На **ws1** запущен сервер: `iperf3 -s -f m`  
- На **ws2** запущен клиент, подключающийся к серверу ws1 (192.168.100.10). Тест выполнен в двух режимах: обычная передача (upload) и обратная (download) с ключом `-R`.

![iperf3 клиент и сервер](Screenshot%202026-03-05%20225624.png)  
![iperf3 продолжение](Screenshot%202026-03-05%20230615.png)  
![iperf3 reverse mode](Screenshot%202026-03-05%20230902.png)

*На скриншотах видно:*  
- **Upload** (клиент → сервер): пропускная способность около **2.7 Gbps**.

- Созданы файлы `/etc/firewall.sh` на ws1 и ws2 с разными стратегиями:  
  ![Содержание firewall.sh для ws1 и ws2](Screenshot%202026-03-05%20233002.png)  
  *На левом скриншоте (ws1) сначала запрещающее правило для ICMP echo-reply, затем разрешающее; на правом (ws2) — наоборот.*

- Запуск файлов:  
  `chmod +x /etc/firewall.sh` и `sudo /etc/firewall.sh` (выполнение команд не отображено на отдельных скриншотах, но результат применения правил виден далее).

- **Разница между стратегиями:**  
  В первом случае (ws1) правило `DROP` для `echo-reply` стоит раньше правила `ACCEPT`, поэтому все ICMP-ответы (ping) будут отброшены. Во втором случае (ws2) правило `ACCEPT` стоит первым, поэтому ping проходит, а `DROP` не срабатывает. Таким образом, ws1 не отвечает на ping, а ws2 отвечает.

- Просмотр правил iptables:  
  ![iptables -L после применения](Screenshot%202026-03-05%20233456.png)  
  *В цепочке OUTPUT видны два правила для ICMP: ACCEPT и DROP. Порядок различается на машинах.*
- Проверка ping с ws1 на ws2 (или наоборот) показывает потерю пакетов:  
  ![ping с потерей](Screenshot%202026-03-05%20233456.png)  
  *4 пакета отправлено, 0 получено.*

## Part 6. Динамическая настройка IP с помощью DHCP

### 6.1. Настройка DHCP-сервера на r2

На роутере **r2** настроен DHCP-сервер для выдачи адресов в сеть `10.20.0.0/26`.  
Для этого отредактирован файл `/etc/dhcp/dhcpd.conf`:

sudo nano /etc/dhcp/dhcpd.conf

В конец файла добавлена следующая конфигурация:

subnet 10.20.0.0 netmask 255.255.255.192 {
    range 10.20.0.2 10.20.0.50;
    option routers 10.20.0.1;
    option domain-name-servers 8.8.8.8;
}

![r2 dhcpd.conf](Screenshot%202026-03-06%20112440.png)

Затем в файле `/etc/default/isc-dhcp-server` указан интерфейс, на котором будет работать DHCP-сервер:

sudo nano /etc/default/isc-dhcp-server

INTERFACESv4="enp0s8"

После этого служба DHCP запущена и добавлена в автозагрузку:

sudo systemctl restart isc-dhcp-server
sudo systemctl enable isc-dhcp-server
sudo systemctl status isc-dhcp-server

Статус службы подтверждает, что она активна и слушает интерфейс `enp0s8`:

![r2 dhcp status](Screenshot%202026-03-06%20113543.png)

### 6.3. Настройка DHCP-сервера на r1 с резервированием адреса для ws11

На роутере **r1** перед началом настройки проверено наличие пакета `isc-dhcp-server`. Пакет не был установлен:

![r1 dpkg isc-dhcp-server](Screenshot%202026-03-06%20114447.png)

Для установки пакета потребовался временный доступ в интернет (через дополнительный NAT-адаптер). После установки пакета конфигурация выполнена аналогично r2, но с добавлением резервирования адреса для ws11 по MAC-адресу.

**MAC-адрес интерфейса ws11** (enp0s3) – `08:00:27:65:64:b8` (виден на скриншотах ws11).  
В файл `/etc/dhcp/dhcpd.conf` на r1 добавлен следующий блок:

subnet 10.10.0.0 netmask 255.255.192.0 {
    range 10.10.0.10 10.10.0.100;
    option routers 10.10.0.1;
    option domain-name-servers 8.8.8.8;
    host ws11 {
        hardware ethernet 08:00:27:65:64:b8;
        fixed-address 10.10.0.2;
    }
}

В файле `/etc/default/isc-dhcp-server` указан интерфейс `enp0s3`. После запуска службы её статус активен:

![r1 dhcp status](Screenshot%202026-03-06%20115359.png)

### 6.6. Использованные опции DHCP

В ходе выполнения задания были применены следующие опции DHCP:

- **`range`** – задаёт пул адресов, которые DHCP-сервер может выдавать клиентам.
- **`option routers`** – указывает шлюз по умолчанию, который будет передан клиенту.

- **`host`** с параметрами `hardware ethernet` и `fixed-address` – резервирует конкретный IP-адрес для клиента на основе его MAC-адреса.

### 7.2. Создание скрипта фаервола на r2

На роутере **r2** создан файл `/etc/firewall.sh` со следующим содержимым:

#!/bin/sh

# Удаление всех правил в таблице filter и nat
iptables -F
iptables -X
iptables -F -t nat
iptables -X -t nat

# Политика по умолчанию - отбрасывать все маршрутизируемые пакеты
iptables -P FORWARD DROP

# Разрешить маршрутизацию всех пакетов протокола ICMP
iptables -A FORWARD -p icmp -j ACCEPT

# Разрешить маршрутизацию TCP трафика (для DNAT и SNAT)
iptables -A FORWARD -p tcp -j ACCEPT

# Включить SNAT для локальной сети 10.20.0.0/26
iptables -t nat -A POSTROUTING -s 10.20.0.0/26 -o enp0s3 -j MASQUERADE

# Включить DNAT для доступа к веб-серверу на ws22 извне (порт 8080)
iptables -t nat -A PREROUTING -i enp0s3 -p tcp --dport 8080 -j DNAT --to-destination 10.20.0.12:80

После применения скрипта:

sudo chmod +x /etc/firewall.sh
sudo /etc/firewall.sh

Проверка правил:

### 7.6. Использованные опции iptables

- **`-P FORWARD DROP`** – установка политики по умолчанию для цепочки FORWARD на DROP.
- **`-A FORWARD -p icmp -j ACCEPT`** – разрешение маршрутизации ICMP-пакетов (пингов).
- **`-A FORWARD -p tcp -j ACCEPT`** – разрешение маршрутизации TCP-трафика.
- **`-t nat -A POSTROUTING -s 10.20.0.0/26 -o enp0s3 -j MASQUERADE`** – включение SNAT для исходящего трафика из сети 10.20.0.0/26 через интерфейс enp0s3.
- **`-t nat -A PREROUTING -i enp0s3 -p tcp --dport 8080 -j DNAT --to-destination 10.20.0.12:80`** – включение DNAT для входящих подключений на порт 8080 интерфейса enp0s3 с перенаправлением на веб-сервер ws22 (10.20.0.12:80).
